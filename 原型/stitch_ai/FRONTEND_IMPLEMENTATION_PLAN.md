# Stitch AI 前端界面实现设计与计划

本文档用于持续推进 `/data/project/ai-study/原型/stitch_ai` 当前保留的 11 个原型界面的微信小程序前端实现。当前阶段只做前端和本地模拟数据，后端接口在前端验收完成后再设计接入。

页面之间的业务流转和模块关系以 `原型/stitch_ai/INFORMATION_ARCHITECTURE.md` 为准；本文档负责开发计划、测试验收和执行记录。

## 1. 当前状态

- 项目类型：微信小程序原生前端，源码目录为 `/data/project/ai-study/miniprogram`。
- 已有页面：`pages/home/home`、`pages/learn/learn`、`pages/checkin/checkin`。
- 已完成基础能力：`pages/home/home` 已实现 `_2` 登录与单次测评入口的核心交互，包括手机号、验证码、游客模式、学段、年级、科目选择。
- 待处理问题：`learn` 和 `checkin` 仍是早期示例页面，不属于 `stitch_ai` 的完整界面体系；后续应按本计划替换或迁移。
- 源码不是 Git 仓库，执行日志需要落到文档或日志目录中，不能依赖 Git 历史回溯。

## 2. 目标范围

本轮大目标是还原 `stitch_ai/_1` 到 `stitch_ai/_11` 的有效前端界面，并建立可反复执行的开发、测试、验收流程。

不在当前阶段做的内容：

- 不接真实登录、短信、测评、学习、报告、导师、拍照识别等后端接口。
- 不上传正式版本。
- 不改变小程序 AppID、基础库和微信开发者工具配置，除非验收流程明确需要。

## 4. 信息架构与导航

推荐保持自定义顶部栏和底部导航，不优先使用原生 `tabBar`。原因是原型中不同页面底栏项、状态和页面高度差异较大，自定义导航更容易贴合视觉稿。完整架构图见 `原型/stitch_ai/INFORMATION_ARCHITECTURE.md`。

核心路径：

1. `_2` 登录与测评入口 -> `_6` AI 学习首页。
2. `_2` 登录与测评入口 -> `_1` 英语测评答题 -> `_3` 测评报告。
3. `_6` AI 学习首页 -> `_4` 专项学习设置 -> `_8` 认读模式或 `_7` 跟读模式 -> `_5` 学习报告。
4. `_3/_6/_11` -> `_9` 知识点掌握 -> `_10` 错题集。
5. `_11` 英语画像我的页 -> 学习任务 `_4`、知识 `_9`、错题 `_10`。

前端导航规则：

- 当前没有后端，所有选择结果通过 `wx.setStorageSync` 或页面 query 临时保存。
- 所有 CTA 必须有可验证行为：页面跳转、状态切换、toast 或本地数据变化。
- 不能只做静态图，答题、筛选、录音态、学习模式选择、表单校验都要有前端交互状态。
- 底部“我的”统一进入 `_11`，不再规划额外我的页。

## 5. 前端设计方案

### 5.1 目录规划

```text
miniprogram/
  app.json
  app.wxss
  components/
    app-top-bar/
    bottom-nav/
    progress-bar/
    metric-card/
    action-card/
    empty-state/
  pages/
    home/home
    dashboard/dashboard
    assessment/exam/exam
    assessment/report/report
    study/setup/setup
    study/recognition/recognition
    study/speaking/speaking
    study/report/report
    knowledge/overview/overview
    mistakes/list/list
    profile/english/english
  utils/
    mock-data.js
    route.js
    storage.js
```

说明：如果微信开发者工具对嵌套页面路径没有问题，优先使用上面的分组路径；如果工具或构建不稳定，则降级为 `pages/assessment-exam/assessment-exam` 这种扁平命名。

### 5.2 样式策略

- 以 `原型/stitch_ai/intelligent_learning_system/DESIGN.md` 为设计 token 来源。
- 全局背景保持 `#f7f9fc`，卡片使用白底、细边框、轻阴影。
- 主要色使用 `#005da7` / `#0060ac`，辅助成功色使用 `#006b58` / `#67f7d5`。
- 页面主容器按 390px 和 429px 两类原型宽度兼容，使用 `rpx` 还原布局。
- 字体使用系统字体栈，不在小程序中依赖 Google Fonts。
- 图标优先使用现有 `miniprogram/assets/prototype` 图片和 CSS 图形；新增图标统一放到 `miniprogram/assets/prototype`，避免散落。

### 5.3 数据策略

前端阶段用本地 mock 数据支撑完整流程：

- `assessmentQuestions`：题目、选项、答案、解析、用户答案、题型。
- `assessmentReport`：总分、正确数、错误数、知识点掌握、建议。
- `studyUnits`：年级、单元、学习模式、单词、句子。
- `speechPractice`：句子、中文释义、分数、反馈、录音状态。
- `knowledgeStats`：词汇、语法、口语、听读写分项和洞察。
- `mistakeItems`：错题分类、掌握度、日期、导师数量。
- `profile`：用户资料、标签、任务、画像指标。

所有 mock 数据应集中在 `utils/mock-data.js`，页面只负责展示和状态变更，为后续后端替换留出清晰边界。

## 6. 开发里程碑

### M0 基线确认

范围：已实现的 `_2` 登录与测评入口。

任务：

- 对 `pages/home/home` 做一次回归，不重写已完成页面。
- 将“登录”和“游客继续”从 toast 改为跳转 `_6` 首页。
- 将“立即开始测评”从 toast 改为跳转 `_1` 答题页。
- 保留登录、验证码、游客、学段、年级、科目选择自测。

验收：

- 登录相关交互不回退。
- 学段切换后年级列表正确变化。
- 登录成功或游客继续能进入 `_6`。
- 开始测评能进入 `_1`。

### M1 前端基础设施

范围：全局结构、公共组件、mock 数据。

任务：

- 更新 `app.json` 页面列表。
- 建立公共顶部栏、底部导航、进度条、指标卡片、操作卡片组件。
- 建立 `utils/mock-data.js`、`utils/route.js`、`utils/storage.js`。
- 整理全局 `app.wxss`，统一卡片、按钮、文字、页面容器基础样式。

验收：

- 所有新增页面可以直接打开。
- 公共组件在至少 3 个页面复用。
- 旧页面样式不因全局样式变化出现明显错位。

### M2 登录后首页与测评闭环

范围：`_6`、`_1`、`_3`，并串联 `_2`。

任务：

- 实现 `_6` AI 学习首页，覆盖今日成就、任务分组、智能提分工具、推荐、AI 助教入口。
- 实现答题页：倒计时、进度条、单选、填空、判断、选中态、上一题/下一题。
- 实现报告页：总分、结果标签、正确/错误题、知识点进度条、咨询表单、错题解析、提升练习。
- 答题完成后生成本地报告数据并跳转报告页。

验收：

- `_2 -> _6` 可跑通。
- `_2 -> _1 -> _3` 可跑通。
- 至少支持 3 种题型状态切换。
- 咨询表单有姓名和联系电话校验。
- 报告页按钮都有前端反馈或跳转。

### M3 专项学习闭环

范围：`_4`、`_8`、`_7`、`_5`。

任务：

- 实现专项学习设置页，支持年级、科目、单元、学习模式选择。

- 实现跟读模式页，支持长按录音态、重录、下一题、模拟评分。
- 实现学习报告页，展示得分、录音播放态、练习详情、返回首页、下一组练习。

验收：

- 从 `_6` 或 `_11` 的学习任务能进入 `_4`。
- 从学习设置能进入认读或跟读。
- 认读选择会更新本地学习状态。

- 学习报告能根据前一页状态展示模拟结果。

任务：

## 7. 可反复执行流程

每次开发按一个 Run ID 执行，格式建议为 `YYYYMMDD-HHMM-范围`，例如 `20260517-frontend-m2`。

### 7.1 开发前检查

```bash
cd /data/project/ai-study
find 原型/stitch_ai -maxdepth 2 -type f | sort
find miniprogram/pages -maxdepth 3 -type f | sort
python3 -m json.tool miniprogram/app.json >/dev/null
```

记录：

- 本轮范围。
- 对应原型编号。
- 计划改动文件。
- 当前阻塞项。

### 7.2 开发

执行顺序：

1. 先读对应 `原型/stitch_ai/_编号/code.html` 和 `screen.png`。
2. 再读将要修改的 `wxml`、`wxss`、`js`、`json`。
3. 按页面最小闭环实现，避免跨里程碑大改。
4. 每个页面完成后补齐本地 mock 数据和跳转。

开发原则：

- 先保证页面可打开，再补交互，再做视觉贴合。
- 交互必须可测，不能只写静态结构。
- 通用结构先抽组件，单页面特殊布局留在页面内。
- 后端字段不要提前固化，只保留 mock 数据与页面状态边界。

### 7.3 静态检查

按实际修改文件扩展执行：

```bash
node --check miniprogram/pages/home/home.js
python3 -m json.tool miniprogram/app.json >/dev/null
find miniprogram/pages -name '*.json' -print -exec python3 -m json.tool {} >/dev/null \;
```

如果新增或修改多个 JS 文件，逐个执行 `node --check`，并把结果记录到执行日志。

### 7.4 模拟器自测

```bash
.tools/bin/wechat-devtools-cli open --project /data/project/ai-study/miniprogram --port 3799 --disable-gpu --trust-project
.tools/bin/wechat-devtools-cli auto --project /data/project/ai-study/miniprogram --auto-port 9420 --port 3799 --trust-project
```

如果缺少自动化依赖：

```bash
npm install --prefix /tmp/miniprogram-automator-run miniprogram-automator
```

每个里程碑至少保留：

- 一个入口路径截图。
- 一个关键交互后截图。
- 一份自动化输出摘要。

截图命名统一放在：

```text
wx_login/screenshots/selftest/stitch-ai-页面名-状态-app.png
wx_login/screenshots/prototype/stitch-ai-页面名-状态-app.png
```

### 7.5 视觉验收

对每个页面检查：

- 页面背景、卡片宽度、圆角、边框、阴影。
- 顶部栏、底部栏、固定按钮位置。
- 标题、数值、标签、按钮文案是否完整。
- 390px 和 429px 原型宽度下是否无横向溢出。
- 长页面滚动到底部后内容和底部操作区不遮挡。

有争议的对齐差异用截图裁剪和像素测量记录，不只凭主观判断。

### 7.6 预览与产物

全量验收通过后执行：

```bash
./scripts/preview-qr.sh
find wx_login -maxdepth 3 -type f | sort
```

保留：

- `wx_login/README.md`
- `wx_login/qr/` 当前预览二维码和信息。
- `wx_login/screenshots/` 最近有效证据。

清理：

- 临时调试裁剪图。
- 失败重试截图。
- 已过期且会干扰当前验收判断的截图。
